# Apaga las Luces (Lights Out) — Design Specification

This document specifies the design and architecture for adding a new "Apaga las Luces" (Lights Out) game to the SourcePlay monorepo.

---

## 1. Requirements

- **Start Menu**: Select difficulty (Fácil, Medio, Difícil, Experto) and specify an optional seed.
- **Difficulties**:
  - **Fácil**: 4x4 grid (16 cells)
  - **Medio**: 5x5 grid (25 cells)
  - **Difícil**: 7x7 grid (49 cells)
  - **Experto**: 9x9 grid (81 cells)
- **Seeded Generation**: A text seed deterministically configures the board generation. If no seed is provided, a random one is generated.
- **Solvability & Optimal Solution**:
  - Puzzles are generated by applying a random set of clicks on an initially all-off grid. This guarantees they are 100% solvable.
  - The game must compute the absolute minimum number of moves (optimal solution) required to solve the generated puzzle.
  - At the end of the game, the player's total move count will be displayed alongside the optimal move count.
- **Action Buttons**:
  - In-game view includes a row of 5 action buttons:
    1.  **Deshacer (Undo)**: Reverts the last move.
    2.  **Rehacer (Redo)**: Re-applies a reverted move.
    3.  **Crear Checkpoint (Save Checkpoint)**: Saves current game state (max 3 slots).
    4.  **Cargar Checkpoint (Load Checkpoint)**: Restores a saved state.
    5.  **Pista (Hint)**: The **rightmost** button. Computes the optimal moves from the current state and highlights the next tile the player should click.
  - Internal control buttons:
    - **Reiniciar (Restart)**: Returns the board to its initial state, resetting moves and history.
    - **Salir al menú (Exit to Menu)**: Exits to difficulty selection (warns if game in progress).
- **Styling**: Favour a minimal *paper* design matching the aesthetics of sliding-puzzle, supporting light and dark themes, using shared design variables.
- **End Game options**:
  - Option to restart the same board, start a new game, or return to the main homepage/selector.

## 3. Core Algorithms

### 3.1 Mathematical Formulation (Linear Algebra over GF(2))
A click configuration of the $N \times N$ grid is represented as a binary vector $x \in \{0, 1\}^{N^2}$, where $x_j = 1$ if cell $j$ is clicked.
Let $s \in \{0, 1\}^{N^2}$ be the state of the lights (1 = on, 0 = off).
Clicking cell $j$ toggles cell $j$ and its adjacent orthogonal neighbors. This corresponds to adding the column vector $A_j$ of the game matrix $A \in \{0, 1\}^{N^2 \times N^2}$ to the state vector:
$$s' = s \oplus A_j$$

Applying a configuration of clicks $y$ to a state $s_{curr}$ transforms the lights to all-off (0) if:
$$s_{curr} \oplus A y = 0 \implies A y = s_{curr}$$

To solve this:
1.  **Particular Solution ($y_0$)**: Using Gaussian elimination over GF(2), we find a particular solution to $A y_0 = s_{curr}$.
2.  **Nullspace ($\ker(A)$)**: At startup or size changes, we calculate the reduced row echelon form (RREF) of $A$ to obtain a basis $\{z_1, z_2, \dots, z_k\}$ of $\ker(A)$.
3.  **Optimal Solution ($y_{opt}$)**: All solutions are given by $y = y_0 \oplus z$ for $z \in \ker(A)$. We iterate over all $2^k$ linear combinations of the basis vectors and select $y_{opt}$ that minimizes the Hamming weight (number of active clicks):
    $$y_{opt} = \operatorname{argmin}_{z \in \ker(A)} \operatorname{weight}(y_0 \oplus z)$$

*Note on Nullspace Dimensions ($k$)*:
- $4 \times 4$: $k = 4$ ($2^4 = 16$ states to check)
- $5 \times 5$: $k = 2$ ($2^2 = 4$ states to check)
- $7 \times 7$: $k = 0$ ($2^0 = 1$ state to check)
- $9 \times 9$: $k = 8$ ($2^8 = 256$ states to check)
Since the maximum number of configurations to check is 256, the solver runs in less than 1ms on the client.

### 3.2 Seeded Generation
1.  **Hash Seed**: Hash the input seed to a 32-bit integer for `mulberry32`.
2.  **Click Selection**:
    - With the PRNG, generate a random binary vector $x_{init} \in \{0, 1\}^{N^2}$. To ensure a good distribution, we can choose to click each cell with a 50% probability, ensuring we don't end up with an all-off board (if so, force at least some clicks).
3.  **Apply Clicks**:
    - Let $s_{start} = A x_{init} \pmod 2$ be the initial board configuration.
    - Compute the optimal solution $y_{opt}$ for $s_{start}$ to determine the exact minimum moves $M_{opt}$ to solve this board.
4.  **Load Board**: Render $s_{start}$ to the user and set the target move count display.

### 3.3 Real-time Hints
1.  When the user clicks the "Pista" (Hint) button, the game reads the current board state $s_{curr}$.
2.  The solver finds the optimal click vector $y_{opt}$ to solve the board from $s_{curr}$.
3.  If $y_{opt}$ is empty (the board is already solved), do nothing.
4.  Otherwise, choose a cell $i$ where $y_{opt}[i] = 1$ (e.g. the first one found) and highlight it on the grid (by adding a flashing CSS class) to guide the player.

## 4. Visual Styles

- **Theme Consistency**: Use variables from `@sourceplay/shared/style.css`.
- **Switch Aesthetics**:
  - Active light (`on`): colored in `--amber` (soft amber glow or solid retro-brutal double border style).
  - Inactive light (`off`): colored in `--paper-deep` with a flatter, inset style representing a turned-off switch.
- **Responsiveness**: The board aspect-ratio is locked at $1:1$ and scaled based on viewport size using `max-width: min(100%, 39vh)`, matching other games.
- **Hint Button**: Hollow star icon (`lucide-star`) positioned as the 5th button in the control row.
